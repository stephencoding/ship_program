# 项目理解、当前进展与规划

## 1. 项目理解

本项目是一个面向海事/航运场景的 AIS 船舶交通流可视化与分析平台。核心目标是通过 Web 地图承载船舶位置、轨迹、交通流、气象水文、测距测面、报告导出和预测推演等能力，服务海事通航选址、航运监控和专业分析。

当前产品定位偏 SaaS 平台，主要用户包括普通用户、专业用户、企业用户和管理员。文档中已定义首页、船舶详情、分析工具、预测推演、用户中心、数据报告等模块。

## 2. 技术理解

当前技术栈为：

- 前端：React + TypeScript + Vite
- 地图：Leaflet + React-Leaflet
- 状态管理：Zustand
- UI/样式：TailwindCSS、Ant Design、lucide-react
- 后端规划：Supabase
- 后续算法服务规划：Python FastAPI，用于交通流预测和分析

整体架构方向是“前端地图应用 + Supabase 数据服务 + 外部 AIS/地图/气象数据源 + 独立 AI 预测服务”。

## 3. 当前进展

### 已完成/已有基础

- 已有完整的 PRD 和架构文档，产品范围、页面模块、API 方向、数据库模型基本明确。
- 前端项目已初始化，基础依赖和构建脚本已配置。
- 已完成主布局、顶部导航、首页地图容器和地图覆盖层。
- 首页已接入 Leaflet 地图，当前中心点为上海港，并展示一个示例 Marker。
- 已实现基础工具面板，包括测距、测面、交通流、气象水文等入口。
- Zustand 已建立全局状态，支持当前工具、选中船舶、侧栏开关等状态。
- 测距工具已有可交互实现，支持地图点击选点、绘线、计算海里距离、右键清除。
- 测面工具已有可交互实现，支持多点绘制区域并估算面积。
- 交通流和气象水文目前已有模拟展示。
- Supabase migration 已建立初始数据模型，包括用户、船舶、轨迹点、分析报告、关注船舶、自定义标记等表。
- `document` 目录已有 PRD、架构文档及 HTML 版本，脚本目录有文档生成相关脚本。

### 当前未完成/占位部分

- `/search`、`/analysis` 等路由仍是占位页面。
- 搜索栏仅输出日志，未接入真实搜索逻辑。
- 地图船舶数据目前是示例点，没有接入真实 AIS 数据或 Supabase 查询。
- 船舶详情页、预测推演页、报告页、用户中心等核心页面尚未实现。
- 登录注册导航存在，但对应路由和认证流程尚未实现。
- 交通流、气象水文仍是 mock 展示，未接入真实分析服务或外部数据源。
- Supabase schema 已有雏形，但前端尚未连接数据层。
- README 仍是 Vite 默认模板，尚未改成项目说明。

## 4. 阶段判断

项目目前处于“文档设计较完整 + 前端原型初步搭建 + 数据库雏形已建立”的阶段。已经能看到产品形态和地图交互方向，但距离可用产品还需要完成数据接入、业务页面、认证权限、分析服务和报告能力。

## 5. 后续规划：Mock First

当前建议采用 **Mock First** 路线：先用结构稳定的 mock 数据把主要产品流程和交互做出来，真实 AIS 数据、Supabase 接入和预测服务后续再替换。前端应通过统一 service 层访问数据，避免组件直接绑定 mock，降低后续切换成本。

### 第一阶段：Mock 数据与最小演示闭环

1. 建立 mock 数据层，覆盖船舶实时位置、船舶详情、历史轨迹、交通流热点、气象水文、报告示例等数据。
2. 建立统一数据服务层，例如 `vesselService`、`trajectoryService`、`analysisService`，组件只依赖 service，不直接依赖 mock 文件。
3. 用 mock 船舶列表替换当前地图单个示例 Marker。
4. 完成搜索逻辑，支持按船名、MMSI、IMO、港口等条件检索 mock 数据。
5. 增加船舶详情页，展示基础信息、实时状态、当前位置和轨迹入口。

### 第二阶段：地图交互与核心业务页面

1. 优化测距/测面交互，例如结束绘制、撤销点、结果保留。
2. 基于 mock 轨迹实现轨迹回放和时间范围筛选。
3. 基于 mock 船舶属性实现船型筛选、图层控制、关注船舶、自定义标记等能力。
4. 完善 `/search`、`/analysis`、`/vessel/:id` 等核心页面，形成可演示的业务闭环。

### 第三阶段：分析、报告与用户流程

1. 基于 mock 交通流数据完善交通流分析 UI 和结果展示。
2. 基于 mock 气象水文数据完善气象图层和信息面板。
3. 实现模拟报告生成、报告详情、报告记录和导出入口。
4. 先做轻量用户流程和页面占位，再逐步接入真实登录、权限角色、我的关注、我的标记等能力。

### 第四阶段：真实数据与服务替换

1. 接入 Supabase 客户端配置，并将 service 层实现从 mock 切换到 Supabase/API。
2. 导入或同步真实/半真实 AIS 数据，替换 mock 船舶位置、详情和轨迹。
3. 完善 Supabase RLS 策略和数据访问权限。
4. 建立独立预测服务接口，将交通流预测、区域分析等能力从 mock 逐步替换为真实算法服务。

## 6. 当前优先建议

建议下一步先做“Mock 数据最小闭环”：mock 船舶数据、地图多船展示、搜索、船舶详情、轨迹回放。这个闭环完成后，平台就可以用于演示和交互验证；真实 AIS 数据和 Supabase 接入后续通过 service 层替换即可。
